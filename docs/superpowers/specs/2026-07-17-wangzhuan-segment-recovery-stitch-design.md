# 网赚片段恢复与手动拼接设计

日期：2026-07-17

## 1. 背景

网赚素材批次进入 `partial_failed` 后，任务管理当前只提示“可在工作台重试拼接或下载可用分段”，但工作台没有承接片段恢复的界面，任务管理也没有展示片段级结果。用户无法判断哪些片段成功、哪些片段失败，也无法执行单片重试、批量重试失败片段、下载成功片段或按自定义顺序拼接。

批次 `wzb_20260717051112_be06` 暴露了更具体的问题：9 个生成任务均在提交前因 `asset_review_pending` 失败，错误为“素材缺少 Seedance assetId，不能使用 S3 URL 直接提交”。此类错误不能通过原请求盲目重试解决，界面必须先区分“可直接重试”和“需要修复素材”。

## 2. 目标

- 在网赚工作台“生成批次”模块之后增加独立的“片段恢复与拼接”模块。
- 批次创建成功后才显示该模块，生成批次之前不占页面空间。
- 展示每个片段的最新状态、缩略图、可播放结果、错误和历次尝试。
- 支持单片重试和一键重试全部失败片段，但不对不可重试错误重复提交。
- 支持上传外部视频替换持续失败的片段。
- 使用同一组选中状态完成“下载选中片段”和“加入拼接队列”。
- 支持选择部分成功片段、拖拽调整顺序和手动开始拼接。
- 每次拼接生成独立版本，不覆盖之前的拼接结果。
- 在拼接前检查解码、画面规格、音频轨和预计时长。

## 3. 非目标

- 不在任务管理页增加恢复操作；任务管理继续负责归档结果和下载，恢复操作统一放在管线工作台。
- 不自动拼接。所有新拼接版本均由用户手动触发。
- 不允许失败或仍在处理的片段加入拼接队列。
- 不在首版提供片段 Prompt、模型或参考素材的高级编辑。
- 不在首版实现跨浏览器或跨设备同步拼接队列草稿。
- 不改变正常批次的既有自动生成、自动后处理、QC 和打包流程。

## 4. 页面位置与显示条件

模块插入 `public/wangzhuan-v2.html` 的“生成批次”模块之后，属于顶部流程条的“4 生成视频和质检”。右侧“后台任务队列、关键提醒、运行与结果”保持现有位置。

显示规则：

- 当前工作台没有 `batchId` 时完全隐藏。
- 批次创建后显示，包括生成中、部分失败、拼接中、QC 和已完成状态。
- 从任务管理恢复批次时，加载详情后直接显示并恢复该批次的模块状态。
- 批次没有片段任务时显示等待态，不伪造空片段。

模块采用纵向三段式：

1. 片段状态。
2. 拼接队列。
3. 拼接版本。

模块顶部显示片段总数、成功数、失败数和重试中数量。计数来自批次详情，不从 DOM 推断。

## 5. 片段状态区

### 5.1 列表结构

片段主列表先按 `branchId + branchVariantIndex` 分组，再按组内 `segmentIndex` 排列。每个分组显示分支/变体名称及成功、失败、处理中计数。拖拽拼接不会改变主列表分组或原顺序。每个片段保持一行，主行展示：

- 勾选框，仅有可用视频结果的片段启用。
- 小缩略图；点击或展开后加载播放器。
- 原片段序号、片段角色或标题、时长和画面比例。
- 最新生成状态。
- 最新尝试次数和时间。
- 简短失败原因。
- 与当前状态匹配的操作。

展开行后展示：

- 成功片段的视频播放器。
- 从 `task_attempts` 读取的历次尝试，按时间从旧到新展示。
- 每次尝试的状态、provider、上游任务 ID、开始/完成时间、错误码和错误信息。
- 需要修复素材时的修复说明和入口。

播放器使用 `preload="metadata"`，折叠状态不批量加载完整视频。

### 5.2 片段可用状态

界面将生成任务状态和片段是否有可用视频分开处理。一个生成任务即使失败，也可能通过“上传替换片段”获得可拼接结果。

片段可用状态包括：

| 状态 | 含义 | 可勾选 | 主操作 |
| --- | --- | --- | --- |
| `ready` | Seedance 片段已生成并可读取 | 是 | 下载 |
| `replacement_ready` | 用户上传的视频已替换该片段 | 是 | 下载、更换替代视频 |
| `running` | 首次生成或重试正在进行 | 否 | 无 |
| `retryable` | 失败但满足直接重试条件 | 否 | 重试 |
| `repair_required` | 原因不会因原样重试而消失 | 否 | 修复素材、上传替换片段 |
| `retry_exhausted` | 已达到最大尝试次数 | 否 | 上传替换片段 |
| `unavailable` | 没有可用结果且无法恢复 | 否 | 查看原因 |

`asset_review_pending`、缺少 Seedance `assetId` 和素材审核未完成必须映射为 `repair_required`。上游超时、明确标记为 retryable 的 provider 失败且未耗尽次数可映射为 `retryable`。资格由后端返回，前端不根据中文错误文本猜测。

### 5.3 单片重试

- 只有 `retryable` 状态展示“重试”。
- 点击后立即禁用该片段的重复操作，并显示排队或提交状态。
- 请求必须携带 `idempotencyKey`，重复点击或网络重放不能产生两次上游提交。
- 重试保留原尝试记录，新尝试追加到历史中。
- 成功后自动刷新缩略图、播放器和可选状态，不自动加入拼接队列。

### 5.4 一键重试全部失败片段

- 按钮先基于后端返回的资格对所有失败片段分类。
- 只提交 `retryable` 且未在处理中的片段。
- `repair_required`、`retry_exhausted` 和已在重试的片段保持原状态。
- 操作结果明确显示“已提交 N 个、需修复 N 个、已耗尽 N 个、处理中 N 个”。
- 没有可直接重试片段时按钮禁用，并引导处理素材问题。

### 5.5 上传替换片段

- `repair_required` 和 `retry_exhausted` 状态提供“上传替换片段”。
- 仅接受支持的本地视频格式。浏览器上传前检查类型、大小和可读取的视频元数据；服务端先写临时文件，再执行完整解码、时长、分辨率、FPS 和音频轨检查，全部通过后才归档为输出。
- 替代视频作为新的 `segment_video` 输出关联到原 generation task 和 script；原失败尝试保持不变。
- 片段主状态切换为 `replacement_ready`，并显示“用户替换”来源。
- 再次上传产生新的替代输出，旧替代输出保留在尝试/来源记录中，但只有当前选定输出进入下载和拼接选择。

## 6. 选择与下载

成功片段的勾选状态同时驱动两个动作：

- “下载选中片段（N）”。
- 拼接队列成员。

规则如下：

- 勾选成功片段时加入拼接队列末尾。
- 取消勾选时从拼接队列移除。
- 在拼接队列移除片段时同步取消主列表勾选。
- 同一原片段只能选择一个当前输出，不允许重复加入队列。
- 勾选可以跨分支/变体，用于一次下载或明确的混合编排；界面必须持续展示每个片段的来源变体。
- 单选和多选都复用现有 `POST /api/wangzhuan/download`，请求使用 `batchIds` 和 `outputIds`；服务端继续返回 ZIP。
- 下载不要求片段通过最终成片 QC，但必须存在可读取的输出文件。包内清单标明这些文件是中间片段而非最终交付成片。

## 7. 拼接队列

### 7.1 队列行为

- 只接收 `ready` 或 `replacement_ready` 的片段输出。
- 默认按用户勾选顺序加入；用户可拖拽调整。
- 每项同时展示“当前拼接顺序”和“原片段序号”。
- 支持单项移除、清空队列和恢复某个拼接版本的队列。
- 允许只选择部分成功片段，不要求所有原片段齐全。
- 允许跨变体选择，但队列必须标记为“混合编排”，不能显示为正常完整成片。
- 至少选择一个片段后才能提交。单片拼接仍生成一个独立版本，便于后处理和版本管理。

### 7.2 队列草稿

首版将队列草稿保存在浏览器 `localStorage`，键必须包含当前用户、项目 key 和 `batchId`。草稿只保存 output ID 顺序和更新时间，不保存视频内容、认证信息或远程签名 URL。

恢复草稿时：

- 丢弃已不存在、已删除或不再可用的 output ID。
- 保留其余 output ID 的原顺序。
- 如果某片段当前输出已被新重试结果替代，不自动换成新输出；界面提示用户选择新结果。
- 显示最近自动保存时间。

此方案不需要数据库迁移，也不承诺跨设备同步。后续如需服务端草稿，必须单独评审数据模型。

### 7.3 拼接前检查

队列变化后，前端使用批次详情里的探测元数据进行快速检查并展示：

- 每个文件是否存在且可读取。
- 是否有完整可解码的视频流。
- 分辨率、像素格式、FPS 和时间基是否可标准化。
- 音频轨是否存在且可标准化。
- 各片段时长和预计总时长。

“开始拼接”请求到达服务端后必须再次执行权威检查，不能信任前端结果。服务端会按队列顺序统一编码参数后拼接；不兼容时返回到具体 output ID 的结构化错误，不创建空版本。

## 8. 手动拼接与版本

### 8.1 新建版本

- 用户点击“开始拼接”后，按队列当前顺序提交 output ID 数组。
- 队列只包含同一 `branchId + branchVariantIndex` 且覆盖该组全部必需 `segmentIndex` 时标记为“完整拼接版”。
- 队列只包含同一拼接组的子集时标记为“部分片段拼接版”。
- 队列包含多个拼接组时标记为“混合编排版”，并在提交前要求用户确认跨变体拼接。
- 每次请求创建新的 stitch version，不替换此前 `stitched_video`，也不复用现有 `retry-stitch` 的 `replaceDerivedOutputs` 行为。
- 同一 idempotency key 重放返回同一版本。
- 运行中显示进度，用户可继续查看片段和其他版本，但不能重复提交同一队列请求。
- 成功后新版本置顶；失败版本保留失败原因并允许用原队列再次创建新版本。

### 8.2 版本内容

每个版本保存并展示：

- 版本号和可编辑名称。
- 完整版、部分版或混合编排版标记。
- 使用的 segment output ID、原片段序号和最终顺序。
- 创建者和创建时间。
- 总时长、尺寸、音频状态和拼接检查结果。
- QC 状态、预览和下载入口。

segment output ID 顺序一经创建不可修改。调整顺序必须复制回队列并生成新版本。

### 8.3 版本操作

- “复制队列”用该版本保存的 output ID 顺序替换当前队列，并同步主列表勾选。
- “预览”打开该版本播放器。
- “下载”复用现有按 output ID 打包能力。
- “重命名”只修改显示名称，不修改 output ID 或文件存储路径。
- “删除”只允许删除用户手动创建的拼接版本，必须二次确认；不删除源片段，不影响其他版本，并记录审计事件。

## 9. 数据与接口契约

### 9.1 批次详情扩展

现有 `GET /api/wangzhuan/batches/:batchId` 保持兼容，并为每个 generation task 增加：

```json
{
  "generationTaskId": "gen_be06_002",
  "branchId": "branch_1",
  "branchVariantIndex": 1,
  "segmentIndex": 2,
  "segmentRole": "product_proof",
  "availability": "repair_required",
  "retryEligibility": {
    "status": "repair_required",
    "reasonCode": "asset_review_pending",
    "message": "素材缺少 Seedance assetId",
    "attempts": 1,
    "maxAttempts": 2
  },
  "attemptHistory": [],
  "availableOutputs": []
}
```

`attemptHistory` 从现有 `task_attempts` 表读取。`availableOutputs` 只返回与该 task/script 关联的可用 `segment_video` 输出，包括 Seedance 结果和用户替代结果。

手动拼接输出继续作为 `stitched_video` 出现在批次 `outputs` 中，并增加兼容字段。`stitchKind` 只允许 `complete`、`partial` 或 `mixed`：

```json
{
  "stitchVersion": 2,
  "stitchKind": "mixed",
  "segmentOutputIds": ["out_01", "out_04", "out_06"],
  "sourceGroups": ["branch_1:1", "branch_1:2"],
  "createdBy": "lucy",
  "compatibilityStatus": "pass"
}
```

### 9.2 新增操作接口

拟新增以下批次内操作：

| 方法与路径 | 用途 |
| --- | --- |
| `POST /api/wangzhuan/batches/:batchId/tasks/:taskId/retry` | 单片重试 |
| `POST /api/wangzhuan/batches/:batchId/tasks/retry-failed` | 重试全部符合资格的失败片段 |
| `POST /api/wangzhuan/batches/:batchId/tasks/:taskId/replacement` | multipart 上传替代视频 |
| `POST /api/wangzhuan/batches/:batchId/stitch-versions` | 按 output ID 顺序创建手动拼接版本 |
| `PATCH /api/wangzhuan/outputs/:outputId` | 重命名手动拼接版本 |
| `DELETE /api/wangzhuan/outputs/:outputId` | 删除手动拼接版本 |

重试和新建拼接版本请求必须带 `idempotencyKey`。批量重试返回逐 task 结果，不因一个片段需要修复而回滚其他可重试片段。

服务端根据选中 output 关联的 script 计算 `stitchKind` 和 `sourceGroups`，不接受前端自行声明“完整”或“部分”。跨组请求必须显式带 `confirmMixed: true`，否则返回校验错误。

手动拼接请求形状为：

```json
{
  "idempotencyKey": "stitch-version-...",
  "segmentOutputIds": ["out_01", "out_04", "out_06"],
  "confirmMixed": true
}
```

数组顺序就是最终拼接顺序。服务端必须验证所有 output 属于当前用户、项目和批次，并且每个 output 都是可读取的片段结果。

### 9.3 现有能力复用

- `retryFailedGenerationTask()` 可作为单片重试的底层入口，但公开手动重试时必须把 retry metadata 标记为用户触发，不能继续固定为 `automatic: true`。
- 现有 `retry-stitch` 仅用于整批拼接/后处理重跑，继续保留，不承担手动选片段和版本创建。
- 现有 `POST /api/wangzhuan/download` 已支持 `batchIds + outputIds`，片段和版本下载直接复用。
- 现有 `asset_files`、`workflow_outputs` 和输出探测 metadata 用于替代片段和拼接版本；首版不新增数据库表。
- 手动版本的顺序、类型、创建者和兼容检查信息保存在现有输出资产 metadata 中。若实现时发现现有字段无法保证读取、审计或删除一致性，应停止并单独评审 schema，而不是把信息塞入文件名。

## 10. 前端模块边界

现有 `public/wangzhuan-v2.js` 已接近 4000 行。首版不应继续把恢复模块的状态、渲染和操作全部堆入该文件。

建议新增 `public/wangzhuan-segment-recovery.js`，职责包括：

- 从批次详情构建片段视图模型。
- 管理展开行、选择、下载和拼接队列草稿。
- 调用重试、替换、拼接版本和版本管理接口。
- 独立轮询正在重试或拼接的批次详情。
- 渲染片段状态、拼接队列和版本列表。

`public/wangzhuan-v2.js` 只负责模块装配、当前用户/项目/批次上下文传入和批次详情更新。共享 API、错误展示和 ZIP 下载继续复用 `public/wangzhuan-common.js`。

## 11. 数据流

1. 批次创建或工作台恢复后加载批次详情。
2. 恢复模块根据 tasks、attempt history 和 outputs 形成片段视图模型。
3. 用户对 `retryable` 片段执行单片或批量重试，页面只轮询受影响批次。
4. 用户可上传替代视频；成功后新增可用片段输出。
5. 用户勾选可用输出，同步更新下载计数和本地拼接队列。
6. 队列变化触发本地草稿保存和快速兼容检查。
7. 用户手动提交拼接；服务端权威检查后创建独立版本。
8. 新版本作为输出返回，页面展示预览、下载、复制队列和管理操作。

批次轮询只更新服务端事实。正在拖拽的本地队列不因轮询重排；仅当所引用输出失效时提示并移除无效项。

## 12. 错误与并发处理

- 批次详情刷新失败时保留现有列表，显示非阻塞“刷新失败，正在重试”。
- 单片重试、批量重试、替代上传和拼接分别维护 busy 状态，不全局锁定工作台。
- 服务端资格变化时以最新返回为准；前端遇到 `invalid_state_transition` 后刷新对应片段。
- 重试次数耗尽后立即切换为 `retry_exhausted`，不继续显示重试按钮。
- 替代上传失败保留原失败信息和本地队列，不产生半成品 output。
- 拼接检查失败返回每个 output ID 的原因，并保持队列和排序不变。
- 拼接请求成功但前端响应丢失时，使用 idempotency key 查询或重放同一结果，不能生成重复版本。
- 删除版本时检查其是否仍被打包或 QC 作业使用；冲突时拒绝删除并说明原因。

## 13. 响应式与可访问性

- 桌面沿用当前主内容加右侧后台栏，恢复模块占主内容全宽。
- 窄屏下主行改为多行布局：片段信息、状态和操作不横向溢出。
- 状态不能只依赖颜色，必须包含文字。
- 展开按钮、下载、重试、上传、拖拽替代操作和版本菜单均有可读标签。
- 拖拽排序同时提供键盘可操作的“上移/下移”菜单。
- 动态计数、重试结果、上传进度和拼接状态使用适当的 `aria-live` 区域。
- 固定缩略图、按钮和状态区域尺寸，避免轮询更新造成布局跳动。

## 14. 测试与验收
